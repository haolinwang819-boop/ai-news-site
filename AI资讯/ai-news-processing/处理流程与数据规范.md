# AI 资讯处理流程与数据规范

本文档定义：抓取内容、存储方式、最终展示内容，以及从抓取到邮件/展示的完整处理流程。**后续所有处理相关脚本均放在本目录 `ai-news-processing` 下。**

---

## 一、抓取内容定义

以下字段为「抓取重点」

| 字段 | 类型 | 必填 | 说明 | 用途 |
|------|------|------|------|------|
| `title` | string | 是 | 资讯标题 | 展示、分类、去重 |
| `url` | string | 是 | 原文链接，唯一标识 | 展示、去重 |
| `source` | string | 是 | 来源名称（如「机器之心」「OpenAI Blog」） | 展示、优先级 |
| `published_time` | string (ISO8601) | 是 | 发布时间 | 展示、排序、时间范围过滤 |
| `content` | string | 是 | 正文 | 推送的正文/图/视频 |
| `image_url` | string | 否 | 配图 URL | 可选展示 |
| `priority` | int | 否 | 不由抓取提供，由标准化节点大模型根据来源与内容判断 0/1/2/3 | 排序、展示 |

---

## 二、存储方式

### 2.1 用什么结构，存在哪里

**存储结构**

- **日报结果**：每个自然日一个 JSON 文件，结构见 2.2。字段仅包含最终展示所需。

**存储位置与形态**

- **默认：本地目录。** 路径为项目内 `ai-news-processing/data/`（如 `digests/YYYY-MM-DD.json`）。
- **若需要多机共享、备份或将来做 Web 展示**：可把同一套目录同步到云存储（如阿里云 OSS、AWS S3、OneDrive 等），程序仍按「读/写本地路径」写，由同步工具或挂载负责上传；也可把运行环境直接放在一台云服务器上，数据仍写在该机磁盘。是否上云按需求定，不是必须。

**本地目录结构示例：**

```
ai-news-processing/
├── 处理流程与数据规范.md
├── data/
│   └── digests/            # 日报 JSON，如 2025-03-04.json
├── prompts/
└── scripts/
```

### 2.2 日报结果格式（digests）

- **文件名**：`YYYY-MM-DD.json`
- **编码**：UTF-8。
- **单文件结构**：

```json #推送封面处的结构
{
  "date": "2025-03-04",
  "total_count": 42, #今天共有多少条推送
  "categories": {
    "image_video": [ /* 条目列表 */ ],
    "llm": [ ],
    "breakout_products": [ ],
    "hot_news": [ ],
    "product_updates": [ ]
  } #点开可以具体的分类目录内的文章
}
```

- **单条条目结构**（与「最终展示内容」一致）

### 2.3 单条条目存储格式（写入 digests 的 item）

```json
{
  "title": "string",
  "url": "string",
  "source": "string",
  "published_time": "2025-03-04T00:00:00+00:00",
  "category": "image_video | llm | breakout_products | hot_news | product_updates",
  "priority": "string",
  "content": "string",
  "image_url": "string | null",
  "display_title": "string",
  "summary": "string",
  "key_points": ["string", "string", "string"]
}
```

- `category` 与现有日报五板块一致：`breakout_products`、`hot_news`、`llm`、`image_video`、`product_updates`。
- `priority`：0 重磅 / 1 重要 / 2 日常 / 3 其他。

## 三、最终展示内容（Display）

以下字段会出现在**邮件正文**与**Web 展示**中；未列出的抓取/中间字段仅用于处理，不展示。

| 字段 | 展示方式 |
|------|----------|
| `display_title` | 英文重写标题，不直接复用原始 source title |
| `url` | 标题上的链接，点击跳转原文 |
| `source` | 来源文案，展示为单一来源标签（如 `Anthropic website`、`The Wall Street Journal`、`X @karpathy`） |
| `published_time` | 可选：显示相对时间或日期 |
| `summary` | 英文摘要段落，先解释发生了什么，再解释为什么重要 |
| `key_points` | 3 个英文 bullet point，帮助读者快速理解重点 |
| `image_url` | 可选：配图 |

说明：
- `title` 继续作为原始来源标题保留，用于追溯与内部处理
- 面向用户的展示统一使用 `display_title`、`summary`、`key_points`
- 邮件与网页中的可见文字应为英文，即使源内容是中文
- 若条目属于 `AI黑马产品`，`summary` 与 `key_points` 需要优先提炼产品的特色、优势、创新点、工作流价值，而不是只写“这个产品是做什么的”

## 四、完整处理流程（Pipeline）

### 4.1 总体流程

```
抓取 → 标准化 → 分类 → 去重 → 反思（可选）→ 输出日报结构 → 存储 → 邮件/Web 展示
```

- **抓取**：由 `ai-news-aggregator` 按「抓取内容定义」输出抓取条目
- **标准化**：统一字段与长度限制
- **分类**：按板块定义将每条归入 `breakout_products` / `hot_news` / `llm` / `image_video` / `product_updates`，可带 `sub_tag`。
- **去重**：语义/标题/URL 去重，合并同事件多条为一条，保留最佳来源或最新一条。
- **反思**：检查错分、漏并、同条多板块等，可循环回分类或去重（建议最多 1～2 轮）。
- **输出**：生成符合「单条条目存储格式」的 `categories` 结构，并补齐展示所需的英文编辑字段。
  - 对 `AI黑马产品`，生成展示字段时应优先使用产品页、更新日志、官方说明中的差异化信息，确保摘要和 bullet points 体现产品亮点而非泛功能描述。
- **存储**：写入 `data/digests/YYYY-MM-DD.json`
- **展示**：读取 digest，按「最终展示内容」生成邮件 HTML 或 Web 页面。

这不是一次性的展示优化，而是**每次日更运行都必须执行的默认步骤**。也就是说，daily run 输出的 digest 默认就应包含 `display_title`、`summary`、`key_points`，后续邮件与网页直接消费这些字段。

### 4.2 实现方式

- **编排**：LangGraph
- **节点与 prompt**：每个节点对应独立小 prompt（可放在 `prompts/`）：
  - 标准化（统一字段、格式）
  - 分类（引用板块定义）
  - 去重（同事件/同产品合并规则，保留更权威的来源）
  - 反思（只输出问题与建议，由逻辑决定是否回流）
- **状态**：图内共享状态包含 `input_items` → `normalized_items` → `categorized_items` → `deduped_items`，反思结果决定是否回到分类/去重。
- **脚本位置**：所有上述节点的实现脚本、LangGraph 图定义、工具函数等，均放在本目录（如 `scripts/`、`prompts/`）。

### 4.3 与现有邮件模块的衔接

- 存储后的 `data/digests/YYYY-MM-DD.json` 中 `categories` 结构与现有 `EmailSender.send(categorized_items)` 所需一致。
- 将 JSON 中每条转为现有 `NewsItem`（或兼容字段），即可直接调用现有 `email_sender` 发送；邮件中只渲染「最终展示内容」所列字段。

## 五、板块定义（与分类节点一致）

| 板块 key | 展示名称 | 说明 |
|----------|----------|------|
| `breakout_products` | AI黑马产品 | 仅收录 Product Hunt 当天 AI 榜前 8，以及极少数明确爆火的新 AI 应用产品。必须是新产品，不是模型上新，也不是老产品功能更新。 |
| `hot_news` | AI热点资讯 | 政策、并购、融资、诉讼、芯片、公司高管流动、重大行业事件。不要把博主观点放进来。 |
| `llm` | AI基模 | 模型发布、版本更新、推理能力、记忆模式、协同模式、开源模型、模型平台能力升级。 |
| `image_video` | AI多模态 | 生图、生视频、音频、视觉理解、多模态模型与多模态产品能力更新。 |
| `product_updates` | AI热门产品更新汇总 | 头部 AI 产品的新功能、新模式、新工作流更新，如 Character AI、Lovart、Notion、GitHub Copilot、Superhuman 等。 |

分类与反思节点均以本表为准；子标签（如「生图」「生视频」）可在本目录下单独文档或 prompt 中扩展。

## 六、文档与脚本约定

- **本目录** `ai-news-processing`：存放本文档、所有处理流程相关脚本、prompts 与配置。
- **数据与输出**：统一使用 `ai-news-processing/data/` 下的 `digests/`
- **格式与展示**：以本文档第三节「最终展示内容」与第二节「单条条目存储格式」为唯一约定；新增展示字段或存储字段时同步更新本文档。
